# Отчёт о тестировании плагина Node2Code

**Дата тестирования:** 1 июня 2026 г.

## 1. Executive Summary (Краткая сводка)

Плагин успешно демонстрирует базовые возможности визуального программирования и трансляции графа в код. Однако на данный момент двусторонняя конвертация заблокирована критическими ошибками в модуле парсера C# (ошибки служебной обёртки) и невозможностью доступа к подпространствам (Subspaces). Также выявлены архитектурные недочёты в генерации кода при работе с переменными, требующие рефакторинга ноды `Values`.

## 3. Детальное описание дефектов

### 🔴 Блокирующие ошибки (Blockers)

**[N2C-001] Парсер выдаёт синтаксические ошибки на валидном C# коде**

* **Спецификация:** "Graph is always derived from generated C# code. Load reconstructs the graph."
* **Описание:** При попытке парсинга корректных статических методов (например, вычисление факториала или класса с двумя методами `Main` и `MyMain`) парсер падает с ошибками компиляции.
* **Фактический результат:** Выводятся ошибки `} expected (10:6 (служебная обёртка))` и `Type or namespace definition... expected (14:1)`. Это указывает на то, что код некорректно вставляется в невидимую служебную обёртку (wrapper) перед парсингом, из-за чего ломается баланс фигурных скобок.
* **Ожидаемый результат:** Код должен успешно преобразовываться в AST-дерево и визуализироваться на холсте (Canvas).

**[N2C-002] Недоступны подпространства (Subspaces) узлов управления потоком**

* **Спецификация:** "Subspaces: isolated inner canvases... Access via buttons in the canvas top‑left."
* **Описание:** Кнопки в левом верхнем углу для перехода в подпространства сложных нод (`For`, `While`, `If`) не реагируют на нажатие.
* **Фактический результат:** Невозможно протестировать, отредактировать или просмотреть логику внутри циклов и условий.
* **Ожидаемый результат:** Нажатие на кнопку должно переключать контекст холста на соответствующее подпространство (например, *Declaration*, *Boundary*, *Step*, *Body* для цикла `For`).

### 🟠 Ошибки высокого приоритета (High Priority)

**[N2C-003] Некорректная трансляция мутации переменных в код**

* **Описание:** Присвоение значений переменным через `inputValue` + `Exec` генерирует избыточный и семантически неверный код.
* **Фактический результат:** В сгенерированном коде появляются сбросы значений (`result = 0; i = 0;`), множественные операции в шаге цикла `i = 0, i = (i + 1)` и мёртвый код после оператора возврата (`return result; result = 0;`).
* **Рекомендация по архитектуре (Feature Request):** Текущая концепция (создание переменной через именование Literal-ноды) перегружает одну ноду логикой чтения и записи. Рекомендуется разделить работу с переменными на независимые ноды:
1. **Variable Getter**: Только Data-порт `output`.
2. **Variable Setter**: Порты `execIn`, `execOut`, `inputValue`.



**[N2C-004] У ноды `Else` отсутствует порт `execOut**`

* **Спецификация:** Узлы выполнения (Exec) должны обеспечивать последовательный контроль.
* **Описание:** Нода `Else` имеет подпространство `Body`, но после выполнения ложной ветки поток управления обрывается, так как у ноды `Else` нет выходного порта для продолжения программы.
* **Фактический результат:** Невозможно продолжить выполнение метода после завершения блока `Else`.
* **Ожидаемый результат:** Нода `Else` должна иметь порт `execOut`, который передает управление дальше по графу после выполнения своего `Body`.

### 🟡 Ошибки среднего и низкого приоритета (Medium & Low)

**[N2C-005] Динамическое (несвоевременное) появление портов `exec` у ноды Literals**

* **Спецификация:** Для записи в переменную требуется подать сигнал на порт `Exec`.
* **Описание:** При создании ноды `Literals` и переводе её в режим "Переменная" (ввод имени), порты `exec` отсутствуют и появляются только после принудительного подключения других связей.
* **Фактический результат:** Ухудшение UX, неочевидное поведение графа.
* **Ожидаемый результат:** Как только нода `Literals` получает имя (переходит в режим переменной), порты `execIn` / `execOut` (или их аналоги для записи) должны немедленно отрисовываться на холсте.

**[N2C-006] Отсутствие ноды `Return` в правой панели**

* **Спецификация:** "Right Panel ... Flow/Debug groups". "Return – ports: execIn, value...".
* **Описание:** Ноду `Return` невозможно создать через панель справа, хотя она доступна через контекстное меню.
* **Фактический результат:** Нарушение консистентности UI.
* **Ожидаемый результат:** Нода `Return` должна находиться в группе "Flow/Debug" в правой панели при нахождении на холсте метода.

---

**Заметка для команды разработки:** Ошибка обёртки в парсере (N2C-001) выглядит как проблема формирования строки перед передачей её в Roslyn (или другой анализатор). Стоит проверить, не добавляет ли парсер лишние закрывающие скобки или не теряет ли модификаторы класса при внедрении пользовательского кода в шаблон. Идея с разделением переменных на Getter и Setter (N2C-003) — отличная, это классический паттерн для визуального программирования (как в Unreal Engine Blueprints), который сразу решит проблемы со "спамом" нулями в генераторе кода.
