# Архитектура расширяемого Visual Scripting плагина — объяснение для разработчиков

## 1. Общая концепция

### 1.1. Отказ от `NodeType` enum
Раньше тип ноды жёстко кодировался в перечислении. Это означало, что при добавлении новой ноды нужно было править `NodeType.cs`, а затем десятки `switch` по всему коду (парсер, генератор, исполнитель, редактор).

**Новая архитектура:**  
Тип ноды идентифицируется **строкой** (`TypeId`), а все метаданные о ноде (название, категория, способ генерации кода, способ выполнения, соответствие методам C# и т.д.) хранятся в **реестре нод** (`NodeRegistry`). Реестр заполняется автоматически при запуске: он сканирует сборки, находит классы, помеченные атрибутом `[VisualScriptingNode]`, и регистрирует их.

**Преимущества:**
- Добавление новой ноды: создаёте класс, вешаете атрибут — и всё.
- Парсер больше не знает о конкретных типах нод, он спрашивает реестр: «какая нода соответствует вызову метода `Mathf.Max`?».
- Генератор кода получает из реестра информацию, как генерировать выражение для этой ноды (оператор, вызов метода и т.д.).
- Исполнитель (рантайм) также получает нужную логику выполнения (либо встроенный `RuntimeExecutor`, либо стандартную обработку).

### 1.2. Двунаправленная конвертация код ↔ граф
Система должна не только создавать граф из кода, но и генерировать код из графа. При этом **пользовательские методы** (с объявлением и телом) также должны превращаться в графы и обратно.

- **Парсер** читает C# код и создаёт `GraphData`, содержащий `NodeData` для каждой инструкции/выражения.
  - Если встречается **вызов метода** — парсер ищет в реестре зарегистрированную ноду с таким именем. Если не находит — создаёт `MethodNode` (заглушку).
  - Если встречается **объявление метода** (со своим телом) — парсер создаёт `MethodNode` и рекурсивно парсит его тело в отдельный `GraphData` (подграф).
- **Генератор** берёт `GraphData` и превращает обратно в C# код.
  - Для обычных нод (математика, сравнения) использует информацию из реестра (оператор, формат вызова).
  - Для `MethodNode` генерирует полноценный метод с сигнатурой и телом, полученным из подграфа.
  - Для `ReturnNode` внутри метода генерирует `return`.
- **Исполнитель** (рантайм) может выполнять граф непосредственно в Unity Editor (для отладки) или через `CSharpProcessRunner` (запуск сгенерированного кода в отдельном процессе).

### 1.3. Ключевые абстракции
| Класс | Назначение |
|-------|------------|
| `NodeData` | Данные одной ноды (тип, значение, связи, подграфы и т.д.). Сериализуется. |
| `GraphData` | Контейнер для `NodeData` и `EdgeData`. |
| `NodeDefinition` | Метаданные о типе ноды: `TypeId`, редакторский класс, соответствие методу, оператор, делегаты генерации/выполнения. |
| `NodeRegistry` | Статический реестр, хранит все зарегистрированные `NodeDefinition`. |
| `VisualScriptingNodeAttribute` | Атрибут, которым помечаются классы редакторских нод для автоматической регистрации. |
| `CustomBaseNode` | Базовый класс для всех редакторских нод (GraphProcessor). |
| `MethodNode` | Специальная нода, представляющая пользовательский метод (имеет параметры, возвраты, подграф-тело). |
| `ReturnNode` | Служебная нода внутри подграфа метода, обозначает точку возврата. |

## 2. Компоненты и их взаимодействие

### 2.1. Реестр нод (`NodeRegistry`)

**Регистрация:**
- При старте редактора (или рантайма) вызывается статический метод `AutoRegister`.
- Он обходит все сборки, ищет классы с атрибутом `[VisualScriptingNode]`, создаёт `NodeDefinition` и добавляет в реестр.
- Для каждой ноды может быть указан:
  - `FullMethodName` — если нода соответствует статическому методу (например, `Mathf.Max`).
  - `OperatorSymbol` — если нода соответствует бинарному оператору (`+`, `-`, `==` и т.д.).
  - `MinArgCount` / `MaxArgCount` — для поддержки перегрузок методов.
  - Делегаты `CodeGenerator` и `RuntimeExecutor` (опционально) для кастомной генерации/выполнения.

**Использование:**
- **Парсер:** при встрече вызова метода `Mathf.Max(10, 20)` парсер получает полное имя `"Mathf.Max"`, количество аргументов `2`, вызывает `NodeRegistry.GetMethodCandidates` и получает список подходящих определений. Если ровно одно — создаёт ноду с `TypeId = def.TypeId`.
- **Генератор:** имея `NodeData` с `TypeId = "MathAdd"`, получает `NodeDefinition`, проверяет `IsOperator` и использует `OperatorSymbol` для генерации `a + b`.
- **Исполнитель:** для ноды `MethodNode` (или любой другой) может использовать `RuntimeExecutor` из определения, если он задан.

### 2.2. Модель данных `NodeData`

`NodeData` — это универсальное представление ноды, которое хранит:
- `Id` — уникальный идентификатор (строка).
- `TypeId` — строковый идентификатор типа (например, `"MathAdd"`, `"MethodNode"`).
- `Value` / `ValueType` — для литералов.
- `VariableName` — если нода связана с переменной.
- `Parameters` — список описаний входных параметров (для `MethodNode`).
- `Returns` — список описаний возвращаемых значений (для `MethodNode`).
- `MethodBody` — подграф, содержащий тело метода.
- `ConditionSubGraph`, `BodySubGraph`, `InitSubGraph`, `IncrementSubGraph` — для flow-нод (if, for, while).
- `ExpressionOverride` — исходный текст выражения (для passthrough).

**Важно:** все подграфы — это тоже `GraphData`, они могут быть вложенными сколько угодно раз.

### 2.3. Парсер (`RoslynCodeParser`)

**Как он теперь работает без `switch`:**
- Разбирает синтаксическое дерево C#.
- Для каждого выражения (`BinaryExpression`, `InvocationExpression`, `LiteralExpression` и т.д.):
  - Пытается определить соответствующий `TypeId` через реестр:
    - Для бинарных операторов — получает оператор (+, -, * и т.д.) и вызывает `NodeRegistry.GetOperator`.
    - Для вызовов методов — получает полное имя метода и количество аргументов, ищет кандидатов.
  - Создаёт `NodeData` с правильным `TypeId`.
  - Устанавливает связи между аргументами и портами (имена портов берутся из `ParameterInfo` определения).
- Специальная обработка:
  - Если метод не найден в реестре — создаётся `MethodNode`-заглушка.
  - Если левая часть присваивания — кортеж, парсер анализирует его и заполняет `Returns` у вызываемого метода.
  - Если встречается объявление метода (вне основного тела) — парсит его в `MethodNode` с подграфом.

**Как парсер обрабатывает тело метода:**  
Внутри `ParseMethodDeclaration` он создаёт новый экземпляр `GraphData`, временно подменяет текущий граф, а затем проходит по операторам тела метода так же, как по основному блоку. При этом все локальные переменные метода становятся видимыми внутри подграфа. При встрече `return` создаётся `ReturnNode`.

### 2.4. Генератор кода (`SimpleCodeGenerator`)

**Принцип работы:**
- Проходит по графу в топологическом порядке, генерируя выражения.
- Для каждого узла:
  - Если у узла есть `VariableName` и он не является `MethodNode` с подграфом — генерирует присваивание или объявление переменной.
  - Для выражений (математика, вызовы методов) — вызывает `EmitExpr`, который использует реестр.
- Кроме основного кода, генератор отдельно обрабатывает все `MethodNode` с непустым `MethodBody`: для каждого генерирует метод (сигнатура + тело) и вставляет перед основным кодом.

**Генерация тела метода (`GenerateMethodBody`):**
- Создаёт временный генератор для подграфа.
- Объявляет локальные переменные для параметров (чтобы на них можно было ссылаться внутри).
- Вызывает стандартный алгоритм генерации цепочки инструкций (с поддержкой `ReturnNode`).
- После завершения проверяет, был ли сгенерирован `return`. Если нет — добавляет `return defaultValue;`.

**Поддержка кортежей:**  
Если `MethodNode` возвращает несколько значений, генератор создаёт временную переменную для хранения кортежа, а затем распаковывает её в левые части присваивания.

### 2.5. Исполнитель (рантайм)

**`GraphRunner`** — выполняет граф в Unity (синхронно или как корутину).
- Запускает корневые ноды (без входящего `execIn`).
- Вычисляет значения через `NodeExecutor`.
- Поддерживает `ReturnNode`: при его выполнении устанавливает флаг возврата и немедленно выходит из подграфа.

**`NodeExecutor`** — выполняет отдельную ноду.
- Для ноды с `TypeId` проверяет, есть ли в реестре `RuntimeExecutor`. Если есть — вызывает его.
- Для `MethodNode` с подграфом создаёт новый `GraphRunner`, передаёт ему параметры, запускает подграф и возвращает результат.
- Для стандартных нод (математика, сравнения) использует встроенную логику (но в идеале тоже вынести в реестр).

**Корутины:**  
Если у `MethodNode` флаг `IsCoroutine == true`, то в рантайме такой метод должен возвращать `IEnumerator`. При генерации кода сигнатура метода изменяется, а `GraphRunner` предоставляет отдельный метод `RunCoroutine`, который обходит граф и на каждой ноде, требующей ожидания (`WaitForSeconds`), делает `yield return`.

### 2.6. Редакторские ноды (GraphProcessor)

**`CustomBaseNode`** — основа всех визуальных нод.
- Содержит поле `NodeId` (совпадает с `GUID`).
- Поле `variableName`.
- Методы `InitializeFromData` и `ToNodeData` для сериализации.
- Виртуальное свойство `TypeId`.

**`MethodNode` (Editor)** — реализует визуальное представление пользовательского метода.
- Динамические порты (входы и выходы) создаются через `[CustomPortBehavior]` на основе списков `parameters` и `returns`.
- Контекстное меню позволяет редактировать сигнатуру (имя, типы, добавить/удалить параметры и возвраты).
- Двойной клик открывает вкладку с подграфом (телом метода).

**`ReturnNode` (Editor)** — простая flow-нода, которая внутри подграфа метода обозначает возврат.
- Динамические входные порты создаются на основе `Returns` родительского `MethodNode` (определяется через обход графа вверх).

**`VisualScriptingWindow`** — главное окно редактора.
- Использует реестр для создания нод из `NodeData`.
- Управляет вкладками (основной граф и подграфы `MethodNode`).
- Синхронизирует изменения между визуальным графом и `NodeData`.

## 3. Потоки данных (примеры)

### 3.1. Парсинг кода в граф

```
Исходный код:
    int result = Mathf.Max(10, 20);
    Console.WriteLine(result);

Шаги парсера:
1. Parse() оборачивает код в служебный метод.
2. Обходит синтаксическое дерево.
3. Встречает InvocationExpression "Mathf.Max(10,20)":
   - Полное имя = "Mathf.Max", аргументов = 2.
   - NodeRegistry.GetMethodCandidates -> 1 кандидат (MathfMaxNode).
   - Создаёт NodeData с TypeId = "MathfMax".
   - Создаёт литералы 10 и 20 (LiteralInt).
   - Добавляет связи от литералов к портам "inputA" и "inputB" ноды MathfMax.
4. Левая часть присваивания — идентификатор "result". Создаётся LiteralInt с VariableName = "result", вход "inputValue" связывается с выходом MathfMax.
5. Встречает InvocationExpression "Console.WriteLine(result)":
   - Полное имя = "Console.WriteLine", аргументов = 1.
   - Находит зарегистрированную ноду ConsoleWriteLineNode.
   - Создаёт её, связывает аргумент (переменную result) с портом "message".
6. Результат — GraphData с набором NodeData и EdgeData.
```

### 3.2. Парсинг определения метода в граф

```
Исходный код метода:
    int Add(int a, int b) {
        return a + b;
    }

Шаги:
1. Parse() находит MethodDeclarationSyntax.
2. Создаёт NodeData с TypeId = "MethodNode".
3. Заполняет MethodName = "Add", Parameters = [a,int, a], [b,int,b].
4. Returns = [null,int,-1].
5. Для тела метода создаёт новый GraphData.
6. В теле встречает ReturnStatementSyntax с BinaryExpression "a + b".
   - Бинарный оператор + -> находит NodeDefinition "MathAdd".
   - Создаёт ноду MathAdd.
   - Создаёт литералы/ссылки на параметры a и b.
   - Подключает их к MathAdd.
   - Создаёт ReturnNode, связывает выход MathAdd с входом ReturnNode.
7. Сохраняет этот GraphData в MethodBody.
```

### 3.3. Генерация кода из графа

```
Граф содержит MethodNode "Add" с телом и основной код: result = Add(5,3).

Генератор:
1. Сначала генерирует метод Add:
   - Сигнатура: "int Add(int a, int b)"
   - Тело: генерирует из подграфа:
        return a + b;
2. Затем генерирует основной метод:
   - int result = Add(5,3);
   - (остальной код)
```

### 3.4. Выполнение MethodNode в рантайме

```
При выполнении ноды Add из графа:
1. NodeExecutor.ExecuteNode видит TypeId = "MethodNode" и наличие MethodBody.
2. Создаёт новый GraphRunner.
3. Передаёт значения аргументов (5 и 3) как локальные переменные a и b.
4. Запускает Runner.RunWithResult(MethodBody).
5. В подграфе выполняется MathAdd, затем ReturnNode.
6. ReturnNode возвращает ExecutionResult с ReturnValues = [8].
7. GraphRunner передаёт результат обратно в NodeExecutor, который возвращает 8.
```

## 4. Как разработчику добавлять новую ноду

### 4.1. Простая нода (например, Math.Sin)

1. Создайте класс в `Editor/Nodes/Math/SinNode.cs`:
```csharp
using GraphProcessor;
using VisualScripting.Core.Models;
using CustomVisualScripting.Editor.Nodes.Base;

[VisualScriptingNode(TypeId = "MathSin", Title = "Sin", Category = "Math", FullMethodName = "System.Math.Sin")]
public class SinNode : CustomBaseNode
{
    [Input("x")] public float x;
    [Output("result")] public float result;

    public override string TypeId => "MathSin";
    
    protected override void Process()
    {
        result = (float)System.Math.Sin(x);
    }

    public override NodeData ToNodeData()
    {
        var data = base.ToNodeData();
        data.ValueType = "float";
        return data;
    }
}
```
2. Атрибут `FullMethodName = "System.Math.Sin"` говорит парсеру, что вызов `Math.Sin(x)` в коде нужно преобразовать в эту ноду.
3. Генератор по умолчанию (если не задан `CodeGenerator`) будет генерировать вызов `System.Math.Sin(...)`, так как в атрибуте указан `FullMethodName`.
4. Всё! При следующем запуске нода автоматически зарегистрируется.

### 4.2. Нода, которая не соответствует одному методу (например, кортежное присваивание)

Здесь нужно написать свой `CodeGenerator` и, возможно, `RuntimeExecutor`. Для сложных случаев можно использовать стандартный механизм подграфов.

### 4.3. Добавление `MethodNode` вручную (через UI)
- Открыть окно Node2Code.
- Нажать правой кнопкой на пустое место графа → выбрать "Flow/Custom Method".
- Появится нода без портов.
- Через контекстное меню ("Edit Signature") добавить параметры и возвраты.
- Двойной клик откроет подграф, в котором можно нарисовать тело метода (используя обычные ноды и `ReturnNode`).

## 5. Часто задаваемые вопросы (для разработчиков)

**Q: Зачем нужен `NodeRegistry` и атрибуты, если можно было оставить `switch`?**  
A: `switch` заставляет править центральный перечень типов и десятки мест. Реестр + атрибуты — это **открытая архитектура**: любой может добавить свою ноду, не меняя ядро.

**Q: Почему `MethodNode` имеет список `Returns`, а не один тип?**  
A: Для поддержки кортежей (метод может возвращать несколько значений). Каждое возвращаемое значение отображается на отдельный выходной порт.

**Q: Как работает `ReturnNode` внутри подграфа?**  
A: Когда `GraphRunner` выполняет `ReturnNode`, он не продолжает выполнение по `execOut`, а возвращает `ExecutionResult` с заполненными значениями из входных портов. Вызывающий код (метод) перехватывает этот результат и завершает подграф.

**Q: Можно ли добавить ноду, которая не имеет аналога в C# (например, визуальный таймер)?**  
A: Да. Просто создайте класс с атрибутом `[VisualScriptingNode]`, но не указывайте `FullMethodName`. Такая нода будет доступна только через меню добавления нод в редакторе. В генераторе кода для неё нужно будет написать свой `CodeGenerator` (или оставить её как пустую заглушку, которая не генерирует код).

**Q: Как отладить регистрацию нод?**  
A: В `AutoRegister` можно добавить `Debug.Log($"Registered {def.TypeId}");`. Также можно в любой момент вызвать `NodeRegistry.GetAll()` и вывести список.

**Q: Почему нет поддержки `ref`/`out` параметров?**  
A: Это значительно усложняет модель и рантайм. На первом этапе не реализуется, но архитектура позволяет добавить позже (расширив `ParameterInfo` и логику генерации/выполнения).

**Q: Как обрабатываются перегрузки методов?**  
A: В реестре для одного `FullMethodName` может быть несколько определений с разным `MinArgCount`/`MaxArgCount`. Парсер выбирает подходящее по количеству аргументов, а в перспективе можно добавить и сравнение типов.

## 6. Рекомендации по рефакторингу

1. **Начинайте с малого:** сначала реализуйте реестр и атрибуты, затем переделайте генератор кода (без парсера). Убедитесь, что существующие графы (созданные вручную через UI) продолжают работать.
2. **Постепенная замена:** пока не удаляйте `NodeType`, а сделайте параллельную поддержку: при загрузке старого графа, если в `NodeData` есть поле `Type` (enum) и нет `TypeId`, создавайте `TypeId = Type.ToString()`. Это упростит миграцию.
3. **Тестируйте каждую ноду:** после рефакторинга парсера проверьте, что выражения `a + b`, `Mathf.Max(a,b)`, `x.ToString()` превращаются в правильные ноды.
4. **Используйте юнит-тесты для критических компонентов** (реестр, генерация `MethodNode`, парсинг кортежей). Это сэкономит время при отладке.

## 7. Заключение

Новая архитектура полностью отделяет ядро системы от конкретных типов нод. Добавление новой ноды сводится к созданию одного класса и атрибута. Парсер, генератор и исполнитель автоматически подхватывают её благодаря реестру. Пользовательские методы стали полноправными участниками графа, что позволяет создавать иерархические визуальные скрипты.
